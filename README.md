# Cadence (Resolve Table)

Трекер исходящих писем контрагентам и связанной работы: тематики, статусы ответов,
сроки ожидания, воронка переходов, взаимодействия, задачи, документы, диаграммы и
сводка. Вход защищён серверной авторизацией.

Архитектура:

```
браузер ──HTTPS──▶ nginx ──┬─▶  /            статика SPA (dist/)
                           └─▶  /api/  ─────▶ Node-сервер авторизации (server/, :3001)
```

- **Фронтенд** — статический SPA (React + TypeScript + Vite). Данные приложения
  кэшируются в браузере (`localStorage` + `IndexedDB`) для скорости.
- **Бэкенд** — небольшой Node-сервис: проверяет логин/пароль по bcrypt-хешам,
  выдаёт сессию в httpOnly-cookie, и **хранит данные приложения** (письма, задачи,
  взаимодействия, документы) в файле-снимке на сервере.

> 💾 **Данные хранятся на сервере и переживают обновление версии.** При входе
> приложение подтягивает снимок с сервера; изменения (их делает admin) автоматически
> сохраняются обратно. Браузерное хранилище — лишь кэш. Снимок лежит в `DATA_DIR`
> (по умолчанию `server/data/store.json`), который обновление кода не трогает.
>
> Модель — общий набор данных: admin правит, viewer смотрит. Одновременное
> редактирование с двух устройств разрешается по принципу «выигрывает последний».

---

## Возможности

- **Сводка** — стартовый дашборд: ключевые показатели и диаграммы по всем разделам.
- **Письма** — таблица (дата отправки, контрагент, контакт, email/канал, тематика,
  тема, статус ответа, дата ответа, срок ожидания, ответственный, примечание).
  Срок ожидания считается автоматически. Сортировка по столбцам, фильтры, режим
  «Просмотр» с воронкой и Гантом за выбранный период.
- **Стадии воронки** — редактируемый список; переименование каскадно правит записи.
- **Воронка переходов** — по реальной истории смены статусов (включая возвраты) +
  диаграмма Ганта.
- **Взаимодействия** — журнал контактов, не являющихся письмами (встречи, совещания,
  звонки и любой свой тип). Управляемый список типов, карточки с участниками-тегами,
  фильтр по периоду, диаграммы.
- **Задачи** — список со сроками, раздел «Выполнено», месячный отчёт (печать/PDF),
  диаграммы. Активная задача защищена от случайной правки (режим «Изменить»).
- **Документы** — загрузка/хранение/переименование/удаление файлов с привязкой к
  письму, компании и этапу; примечания к документам; просмотр по компаниям/этапам/
  истории; диаграммы.
- **Глобальный поиск** по всем разделам сразу.
- **Экспорт и копии:**
  - «Экспорт в Excel» — все разделы в один `.xlsx`.
  - «Скачать файлы» — все документы одним `.zip`.
  - «🔒 Зашифрованный ZIP» — документы в архив с шифрованием **AES-256** по паролю.
  - «💾 Копия» / «📂 Восстановить» — полная резервная копия (всё, включая файлы) в
    один JSON и восстановление из него.
- **Роли** — `admin` (полный доступ) и `viewer` (просмотр и скачивание).

## Где хранятся данные

**Источник правды — сервер:** снимок всех данных лежит в `DATA_DIR/store.json`
(по умолчанию `server/data/store.json`). Браузерные хранилища ниже — это **кэш**,
который наполняется из серверного снимка при входе.

| Данные | Где (на сервере) | Кэш в браузере |
| --- | --- | --- |
| Письма, стадии, задачи, взаимодействия, типы | `store.json` | `localStorage` |
| Документы: метаданные | `store.json` | `localStorage` |
| Документы: файлы (бинарь) | `store.json` (base64) | `IndexedDB` |
| Выбранный раздел / режим / колонки | — (только UI) | `localStorage` |
| Сессия входа | подпись (JWT) | httpOnly-cookie `cadence_session` |

Сессия не хранится в localStorage — это подписанный токен в защищённой cookie,
недоступной из JS.

## Технологии

Фронтенд: React 18 · TypeScript · Vite 5 · `@zip.js/zip.js` (шифрованный ZIP).
Бэкенд: Node · Express · bcryptjs · jsonwebtoken (JWT в cookie).

## Локальная разработка

Требуется Node.js 18+. Нужны **два процесса**: сервер авторизации и dev-сервер фронта.

```bash
# 1) сервер авторизации
cd server
npm install
cp .env.example .env          # впишите SESSION_SECRET (см. ниже)
npm start                     # http://localhost:3001

# 2) в другом терминале — фронтенд
npm install
npm run dev                   # http://localhost:4173  (/api проксируется на :3001)
```

Сгенерировать секрет для `.env`:

```bash
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

### Учётные записи по умолчанию

| Логин | Пароль | Роль |
| --- | --- | --- |
| `cadmin` | `Cadence!2026` | admin |
| `cview` | `View!2026` | viewer |

**Поменяйте пароли перед публикацией** (см. раздел деплоя). Список пользователей —
в [server/users.json](server/users.json).

## Деплой на сервер — от и до

Цель: nginx раздаёт собранный фронтенд и проксирует `/api` на Node-сервер
авторизации. Понадобится сервер с Node.js 18+, nginx и (желательно) доменом.

### 0. Что куда положим

Договоримся о путях (можно свои — поправьте в конфигах):

```
/var/www/cadence/          ← весь репозиторий
/var/www/cadence/dist/     ← собранный фронтенд (отдаёт nginx)
/var/www/cadence/server/   ← сервер авторизации (Node)
```

### 1. Положить код на сервер

```bash
sudo mkdir -p /var/www/cadence
sudo chown -R $USER /var/www/cadence
git clone <репозиторий> /var/www/cadence      # или scp/rsync исходников
cd /var/www/cadence
```

### 2. Собрать фронтенд

```bash
npm ci
npm run build                 # появится /var/www/cadence/dist
```

### 3. Настроить сервер авторизации

```bash
cd /var/www/cadence/server
npm install --omit=dev

# создать .env
cp .env.example .env
nano .env
```

В `.env` обязательно задайте:

```ini
SESSION_SECRET=<длинная_случайная_строка>     # node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
NODE_ENV=production
PORT=3001                                      # ← порт сервера авторизации (см. «Где поменять порт»)
SESSION_TTL_SECONDS=28800                      # срок сессии, 8 часов
COOKIE_SECURE=true                             # HTTPS есть → true. HTTPS НЕТ → false (см. «Если HTTPS нет»)
DATA_DIR=data                                  # папка с данными (store.json). Лучше путь вне деплоя, напр. /var/lib/cadence
```

Сменить пароли / завести пользователей:

```bash
npm run hash "новый-надёжный-пароль"          # печатает bcrypt-хеш
nano users.json                                # вставьте хеш в passwordHash
```

`users.json` — массив `{ "username", "role": "admin"|"viewer", "passwordHash" }`.

### 4. Запустить сервер как службу (systemd)

`/etc/systemd/system/cadence-auth.service`:

```ini
[Unit]
Description=Cadence auth server
After=network.target

[Service]
WorkingDirectory=/var/www/cadence/server
Environment=NODE_ENV=production
EnvironmentFile=/var/www/cadence/server/.env
ExecStart=/usr/bin/node server.js
Restart=on-failure
User=www-data

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now cadence-auth
sudo systemctl status cadence-auth            # должно быть active (running)
```

Сервер слушает `127.0.0.1:3001` — наружу его публиковать не нужно, только через nginx.

### 5. Настроить nginx

Готовый конфиг с заголовками безопасности — [deploy/nginx.conf](deploy/nginx.conf).
Скопируйте и поправьте `server_name`, пути к сертификатам и `root`:

```bash
sudo cp /var/www/cadence/deploy/nginx.conf /etc/nginx/sites-available/cadence
sudo nano /etc/nginx/sites-available/cadence     # домен, root=/var/www/cadence/dist, порт прокси
sudo ln -s /etc/nginx/sites-available/cadence /etc/nginx/sites-enabled/
sudo nginx -t                                    # проверка конфига
sudo systemctl reload nginx
```

Конфиг включает: редирект 80→443, HSTS, CSP, X-Frame-Options, проксирование `/api`
на сервер авторизации и SPA-фоллбэк.

### 6. Включить HTTPS (рекомендуется)

HTTPS настоятельно рекомендуется: иначе пароль и cookie сессии идут по сети в
открытом виде. С доменом это бесплатно через Let's Encrypt:

```bash
sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d cadence.example.com
```

certbot подставит сертификаты и редирект 80→443. С HTTPS оставьте в `.env`
`COOKIE_SECURE=true` и используйте [deploy/nginx.conf](deploy/nginx.conf).

> Если HTTPS пока нет — это не блокирует запуск, см. раздел ниже.

### 7. Проверить, что всё работает

```bash
curl -I  https://cadence.example.com/                     # 200, отдаётся index.html
curl -s  https://cadence.example.com/api/me               # {"error":"нет сессии"} + 401 — это норма до входа
```

Откройте сайт в браузере → войдите (`cadmin` / ваш новый пароль) → после перезагрузки
страницы вход сохраняется (сессия в cookie). Готово.

## Если HTTPS нет (деплой по http://)

Запуск возможен и без TLS — нужно поменять два момента, иначе вход не заработает
(Secure-cookie не уходит по http):

1. В `server/.env` поставьте:
   ```ini
   COOKIE_SECURE=false
   ```
   (`NODE_ENV=production` и `SESSION_SECRET` оставьте как есть). Перезапустите сервер:
   `sudo systemctl restart cadence-auth`.

2. Используйте HTTP-конфиг nginx [deploy/nginx-http.conf](deploy/nginx-http.conf)
   вместо `nginx.conf` (в нём нет TLS и HSTS, но есть остальные заголовки):
   ```bash
   sudo cp /var/www/cadence/deploy/nginx-http.conf /etc/nginx/sites-available/cadence
   sudo nano /etc/nginx/sites-available/cadence     # server_name (домен/IP), root, порт
   sudo ln -s /etc/nginx/sites-available/cadence /etc/nginx/sites-enabled/
   sudo nginx -t && sudo systemctl reload nginx
   ```

Чем платите за отсутствие HTTPS:

- Пароль при входе и cookie сессии идут по сети **в открытом виде** — их можно
  перехватить в той же сети. Сама проверка пароля и httpOnly/SameSite-защита cookie
  при этом работают.
- **Шифрованный ZIP** документов в браузере по `http://` на не-localhost адресе может
  не работать: он использует Web Crypto, доступный только в защищённом контексте
  (HTTPS или localhost). Остальные функции работают.

Как только появится домен/сертификат — переключитесь на HTTPS (раздел 6) и верните
`COOKIE_SECURE=true`.

## Пользователи и пароли

**Проще всего — через интерфейс.** Кнопка с именем пользователя в правом верхнем
углу открывает «Аккаунт»:

- любой пользователь может **сменить свой пароль** (нужен текущий);
- **admin** видит список пользователей и может **добавлять**, **удалять**, менять
  **роль** и **сбрасывать пароли**. Защита: нельзя удалить себя или последнего
  администратора. Смена роли вступает в силу при следующем входе пользователя.

Изменения сразу пишутся в `server/users.json`. Ручное редактирование (ниже) —
по-прежнему доступно как альтернатива.

Пароли в открытом виде нигде не хранятся. В файле `server/users.json` лежат только
**bcrypt-хеши**. Этот файл **не в git** (его меняют через интерфейс) — при первом
запуске сервер создаёт его из шаблона [server/users.example.json](server/users.example.json),
поэтому обновления кода учётки не затирают. Формат — массив записей:

```json
[
  { "username": "cadmin", "role": "admin",  "passwordHash": "$2a$12$..." },
  { "username": "cview",  "role": "viewer", "passwordHash": "$2a$12$..." }
]
```

- `username` — логин (регистр при входе не важен).
- `role` — `admin` (полный доступ) или `viewer` (только просмотр и скачивание).
- `passwordHash` — bcrypt-хеш пароля (генерируется командой ниже).

### Сгенерировать хеш пароля

Из папки `server/`:

```bash
npm run hash "мой-надёжный-пароль"
# напечатает строку вида: $2a$12$Q9....
```

(то же самое: `node hash.js "мой-надёжный-пароль"`)

### Поменять пароль существующему пользователю

1. Сгенерируйте хеш нового пароля (команда выше).
2. В `server/users.json` замените `passwordHash` у нужного пользователя на новый хеш.
3. Перезапустите сервер авторизации (он читает `users.json` при старте):
   - прод: `sudo systemctl restart cadence-auth`
   - локально: остановите и снова запустите `npm start`.

### Добавить нового пользователя

Добавьте ещё один объект в массив `users.json` и перезапустите сервер:

```json
[
  { "username": "cadmin", "role": "admin",  "passwordHash": "$2a$12$..." },
  { "username": "ivanov", "role": "viewer", "passwordHash": "$2a$12$<хеш_пароля_Иванова>" }
]
```

### Удалить пользователя

Уберите его объект из массива и перезапустите сервер.

> Стартовые учётки (`cadmin` / `Cadence!2026`, `cview` / `View!2026`) **обязательно
> смените** перед публикацией: их пароли указаны в этом README.

## Где поменять порт

Сервер авторизации по умолчанию на **3001**. Если меняете — поправьте в **трёх** местах
(значение должно совпадать):

| Где | Что | Когда нужно |
| --- | --- | --- |
| [server/.env](server/.env.example) | `PORT=3001` | всегда (это реальный порт сервера) |
| [deploy/nginx.conf](deploy/nginx.conf) | `proxy_pass http://127.0.0.1:3001;` | на сервере (прод) |
| [vite.config.ts](vite.config.ts) | `target: 'http://localhost:3001'` | только для локальной разработки |

Порт **самого сайта** (80/443) задаётся в конфиге nginx (`listen`). Порт dev-сервера
фронтенда (4173) — в [vite.config.ts](vite.config.ts) (`server.port`), на прод не влияет.

## Обновление версии (через git)

Готовый скрипт делает всё разом и проверяет живость:

```bash
cd /var/www/cadence
bash deploy/update.sh
```

Он выполняет: `git pull` → `npm ci && npm run build` → зависимости сервера →
`systemctl restart cadence-auth` → проверку `GET /api/health`. (Если служба
называется иначе — поправьте `SERVICE` в начале скрипта.)

Вручную — то же самое:

```bash
cd /var/www/cadence
git pull
npm ci && npm run build                       # пересобрать фронтенд
cd server && npm install --omit=dev           # если менялись зависимости сервера
sudo systemctl restart cadence-auth           # перезапустить сервер
curl -fsS http://127.0.0.1:3001/api/health    # проверить, что сервер поднялся → {"ok":true}
```

### Что переживает обновление

Всё изменяемое состояние — **в `.gitignore`**, поэтому `git pull` и пересборка
его не трогают:

| Состояние | Где | В git? |
| --- | --- | --- |
| Данные приложения | `server/data/store.json` | нет |
| Пользователи и пароли | `server/users.json` | нет (шаблон — `users.example.json`) |
| Секреты/настройки | `server/.env` | нет (шаблон — `.env.example`) |

Сборка пересоздаёт только фронтенд (`dist/`). После обновления всё введённое и все
учётки остаются на месте. Перед крупным обновлением можно дополнительно скачать
копию кнопкой «💾 Копия».

> Совет: задайте `DATA_DIR` (в `server/.env`) на путь **вне** каталога деплоя,
> например `/var/lib/cadence` — тогда данные не пострадают даже при полной
> переустановке из `git clone`. Не забудьте также сохранить `.env` и `users.json`.

## Резервные копии данных

Так как данные живут в браузере, для переноса/бэкапа используйте кнопки в шапке (admin):

- **«💾 Копия»** — выгружает всё (письма, стадии, задачи, взаимодействия, типы,
  документы вместе с файлами) в один JSON.
- **«📂 Восстановить»** — загружает такой JSON обратно, заменяя текущие данные.
- **«Экспорт в Excel»** — читаемый `.xlsx` со всеми разделами (для отчётов).
- **«🔒 Зашифрованный ZIP»** — все документы в архив с паролем (AES-256).
